# NameValueReferable Interface Implementation - TypedDependency Fix

**Issue**: `ClassCastException: TypedDependency cannot be cast to NameValueReferable`
```
at com.deliveredtechnologies.rulebook.FactMap.put(FactMap.java:13)
at domox.dom.rules.TypedDependencyRulesTest.testTDR1_SubjectEntityExtraction(TypedDependencyRulesTest.java:83)
```

**Status**: ✅ FIXED
**Date**: 2026-05-16
**Build Status**: BUILD SUCCESS ✅

---

## Root Cause

RuleBook's `FactMap.put()` method expects values to implement the `NameValueReferable` interface:
```java
public void put(String name, Object value) {
    // value must be a NameValueReferable
    // ClassCastException thrown if it's not
}
```

`TypedDependency` did not implement this interface, causing the cast to fail at runtime.

---

## Solution

Made `TypedDependency` implement the `NameValueReferable` interface by:
1. Adding the import for NameValueReferable
2. Implementing the interface in the class declaration
3. Adding all required methods: `getName()`, `setName()`, `getValue()`, `setValue()`

## How NameValueReferable Works

### Purpose
The RuleBook framework uses `NameValueReferable` to store and retrieve facts in the FactMap by name.

### Registration Flow
```
Test Code
    ↓
facts.put("currentTd", (NameValueReferable) td)
    ↓
RuleBook FactMap.put(String name, NameValueReferable value)
    ↓
Stores as: name → NameValueReferable.getValue()
    ↓
For retrieval: NameValueReferable.getName() returns the registered name
```

### Implementation Notes
- **getName()**: Returns unique identifier for the fact (e.g., "TypedDependency_1")
- **setName()**: Not used for TypedDependency (immutable)
- **getValue()**: Returns self (the TypedDependency object)
- **setValue()**: Not used for TypedDependency (immutable)

## Design Decisions

### Why NameValueReferable Methods Are No-ops
TypedDependency is:
- **Immutable**: Created once, values don't change
- **Auto-named**: IDs are auto-generated by JPA
- **Read-only in RuleBook**: Only getValue() is used by the rule engine

The methods `setName()` and `setValue()` are deliberately no-op implementations because:
1. They're required by the interface
2. They're not needed for TypedDependency's use case
3. The rule engine doesn't call them
4. Immutability is preserved

### Alternative Approaches Not Used
- ❌ Create wrapper class: Would add complexity
- ❌ Remove NameValueReferable casting: Would fail at runtime
- ❌ Use different FactMap method: No alternative method exists
- ✅ Implement interface: Simplest, most direct solution
